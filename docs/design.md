# Code Sandbox MCP — 設計方針・機能ロードマップ

> 位置づけ: これは「Dockerを操作するMCP」ではなく、**AIが最小限のコンテキストで安全にテスト・検証を実行できる基盤**。
> 機能を増やすことが目的ではなく、トークン消費を抑え・推論精度を保ち・人間の最終制御を残すことを優先する。

---

## 0. 基本思想

- **AI-first**: 全部見せない / 生データではなく構造を返す / 段階的に詳細化する。
- **HITL（Human-In-The-Loop）**: 実行はAI、最終判断は人間。危険操作は承認・Dry Run・リスク表示を標準に。
- **クリーン環境優先**: 状態共有による高速化より、再現性・安全性・デバッグ容易性を優先。原則は使い捨て環境。
- **スコープの線引き**: このMCPは「テスト実行サンドボックス」。コード理解 / RAG は別プロダクトとして扱う。
- **組織化プリミティブ = セッション**: すべてのツールは「Docker操作」ではなく **ライブなセッション（コンテナ＋FS＋run履歴を持つ作業空間）に対する操作** として設計する。機能の所属は常に **「それはセッション状態に対する操作か?」** で判定する（Yes＝入れる / 横断的なコード理解はNo＝§1で切る）。

---

## 1. スコープ判断（あえてやらないこと）

線引きを明確にして発散を防ぐ。判定基準は§0の通り **「ライブなセッションに対する操作か?」**。

- **セマンティックコード検索 / `find_symbol` / コードグラフ / ドキュメントグラフ** → **このMCPに入れない。**
  別の関心（コード理解レイヤー）であり、言語別パーサ・インデクサ・埋め込みストアを抱え込むと保守が膨張し、LSP系・ctags・ripgrep・既存code-RAG MCPと丸かぶりする。必要なら**別リポジトリ**に切り出す。
- **`snapshot_container` / `restore_from_snapshot`** → 後回し（クリーン環境思想と矛盾）。
- **`create_network` / `expose_port_temporarily` / `estimate_cost`** → 後回し。Composeで足りる領域はComposeに寄せる。
- **Tool Auto Discovery / Intelligent Test Runner** → 北極星（ビジョン）として保持。下記が固まるまでロードマップには載せない。

---

## 2. セキュリティ（土台 / Phase 0・非機能）

機能ではなく**最初のコミットから入っている前提**。HITL（実行時承認）とは別の「静的ガードレール」。

- 非root実行（`--user`）を既定強制。
- `--privileged` 禁止。
- `/var/run/docker.sock` 等の危険ソケットのマウント拒否。
- ホストマウントはホワイトリスト制限。
- memory / CPU / pids 制限、ネットワークは既定 off。
- 再現性のため **imageはタグでなくダイジェスト固定**（`image@sha256:...`）。

## 3. トークン削減（LLM視点）

両提案は「1回の出力をどう削るか」に寄っている。ここでは**反復ループ全体での削減**を主眼にする。
系統は **(1)状態をサーバに置く (2)全文でなく差分を返す (3)返す前にノイズを削る**。

### 3.1 状態はサーバ側、LLMは「ハンドル」を持つ
LLMが状態（ログ・ファイル・結果）をコンテキストに抱えることが最大の浪費。サーバ側バッファリングを全機能に一般化する。

- **`run_id` を全結果の起点にする。** 「続きのログ」「失敗したテストだけ再実行」を巨大な再投入なしに参照できる。
- **大きな成果物はインライン展開しない。** カバレッジ・生成物・大JSONは `resource://run/123/coverage (1.2MB)` のようにサイズ付きハンドルで返し、LLMが必要と判断したときだけ fetch させる（MCP resource）。

### 3.2 全文ではなく「差分」を返す（反復ループの本命）
- **run間diff**: 前回比で「test_loginがpass、他2件は前回と同一エラー」のように変化点だけ返す。
- **失敗のフィンガープリント＋重複圧縮**: 各失敗をハッシュ化し、再実行時は「run #2と同一失敗」とだけ。同型失敗は `×N` に畳む（スタックトレースのトークン削減に最大効果）。
- **`rerun_failed(run_id)` / 影響範囲の絞り込み再実行**: 失敗分・変更ファイルが影響するテストだけ実行し、返却も小さな差分集合に。
- **コンテンツアドレスな結果キャッシュ**: image＋コマンド＋入力ファイルのハッシュが不変なら再実行せず `cached: true` を返す（計算・トークン両方を節約）。

### 3.3 返す前にデノイズ（トランケートとは別物）
- ANSIカラー・タイムスタンプ・`\r` 進捗バー（pip/npm）を除去。実行時に `CI=true` / `--no-color` / `pip --progress-bar off` 等を強制。
- 同一行の連続は `[×1000] retrying...` に畳む。
- **ライブラリフレームの剪定**: `site-packages` / pytest内部の大量フレームを落とし、ユーザコードのフレーム・例外型・メッセージ・`file:line`・アサーション行に圧縮。
- **成功はほぼ無料に**: all-passなら `{status: ok, passed: 120, duration: 4.2s}` の一行だけ。

### 3.4 補助
- **トークン予算パラメータ**: `max_output_tokens` を渡すと枠に収まるよう要約し「続きはこのハンドル」を付ける。
- **コマンドのバッチ実行**: `[cmd1, cmd2, cmd3]` を1コールで。往復減＝エージェントの再推論ぶんのトークンも減る。

### 原則（必須）
**既定は差分・要約、ただしフルは常にハンドルで取得可能（offset/limitで前後N行を取り直せる逃げ道を必ず残す）。**
フィンガープリント畳み込みや差分返却は、要約が外れるとエージェントが盲目になるため。

## 4. 構造化テスト結果（本丸）

5000行のログではなく結果を返す。AI-firstの核心。

```json
{
  "status": "failed",
  "duration": 12.3,
  "passed": 120,
  "failed": 2,
  "failures": [
    { "test": "test_login", "error": "AssertionError", "file": "auth/login.py", "line": 42 }
  ]
}
```

- 共通スキーマで `export_test_report` を出力。
- **現実的なコスト**: フレームワークごとに reporter 注入（`pytest-json-report` + `--json-report` 等）/ パースの**アダプタ層**が必要で、ここが工数とメンテ負債の中心。
- **v1は pytest / jest / go test の3つを"ちゃんと"対応。** terraform は後追い。
  - pytest: `pytest-json-report` + `--json-report`
  - jest: `jest --json`
  - go test: `go test -json`（標準出力がJSONストリームなのでパーサが比較的軽い）

## 5. Edit/Verify サブシステム（最小編集ループ）

> 位置づけ: **テストが失敗したときに、その場で少しだけ効率よく直して再検証するためだけ** のツール群。汎用エディタでもIDEでもない。§4の構造化失敗（`file:line`付き）を入力に、最小の修正 → 再lint/再test を回すことだけが目的。

**独立性の設計**
- §0のセッションプリミティブ上に乗る**独立サブシステム**（独自インターフェース・独自章）。
- ただし **同一サーバ内に同居**。別サーバにすると、このサーバが所有するエフェメラルコンテナへのハンドルを失い、edit→verifyのループが壊れる。「独立設計・同居デプロイ」が原則。

**コア（これだけでループが閉じる）**
- **`apply_patch`**: unified diff を当てる。ファイル全文の送受信を避け、トークンを1〜2桁削減。
- **`read_file_range`**: `offset` / `limit` で該当箇所だけ読む。失敗の `file:line` の前後のみ。
- **`lint_in_container` / `type_check_in_container`**: 生出力でなく `{file, line, rule, message}` の構造化配列で返す。testの前段で安く潰す。

**二次（あると便利・需要が出てから）**
- `search_in_container`（grep相当・ファイル/行だけ返す） / `format_in_container`（変更ファイル一覧のみ） / `tree_in_container`（depth制限） / `git_diff_in_container`（セッション中の変更追跡）。

**スコープ境界（重要）**
- 対象は **使い捨てサンドボックス内のファイルのみ**。ユーザの実リポジトリ作業ツリーは対象外（filesystem MCP / エディタ統合の領分）。
- `git_reset` / `git_stash` 等の破壊的操作は §7（HITL）対象。
- 「独立サブシステムになった」を口実にツールを増やさない。**コア3点が主役**、二次は後追い。

**理想ループ（最小コンテキスト）**
```
search_in_container → read_file_range → apply_patch → lint/type_check → rerun_failed
```
全文の読み書きで回すと1ループ数千トークン、このフローなら数百トークンで収まる。

## 6. 出力制御（既存提案の確定版）

- `verbose`: `error_only` / `summary` / `full`（既定 `summary`）。
- トランケート＋メタデータ: `{ "shown": 20, "total_lines": 5000, "truncated": true }`。
- ページング: `offset` / `limit`、レスポンスに `next_offset` / `has_more`。
- 終了コード≠0 / stderrありの場合はエラー周辺を多めに含める。

## 7. HITL（実効性のある承認）

「フラグ方式（`requires_human_approval: true`）」はクライアントが善意で従う前提なので弱い。

- **二段階トークン方式（基本）**: `dry_run` が実行予定（例 `docker rm xxx`）＋確認トークンを返し、本実行はトークン無しでは拒否。クライアント協力なしでも効く。
- **elicitation（補助）**: 対応クライアントでは確認UIを返す。ただしクライアント依存なのでトークン方式のフォールバック必須。
- 対象例: DB削除 / データ破壊 / 永続ボリューム削除 / ネットワーク変更。

---

## 8. 人間が把握しやすい仕組み（可観測性 / オーバーサイト）

現状の「studioを横で見る」は生のツール呼び出しを追う形。**人間向けにキュレーションした監視面**を用意する。

- **ローカルWebダッシュボード（localhost限定 / read-mostly / 自動更新）**: 稼働コンテナ・run履歴・pass/fail・承認待ち・リソース使用量を一目で。studio併用の正統な置き換え。
- **人間可読の append-only 実行ジャーナル**: `tail -f ~/.code-sandbox-mcp/journal.log` で「いつ・どのimageで・何を・dry-run/実実行・danger level・結果サマリ」が自然文で流れる。UIが無くても把握度が段違い。
- **承認キュー＋ワンクリック Approve/Reject**: §7の二段階トークンと連動。「祈る」から「押す」へ。
- **プッシュ通知（OS通知 / Webhook）**: 危険操作・失敗閾値超え・長時間実行のときだけ人を呼ぶ。**「常に見ている」必要をなくす**のがstudio併用に対する本質的な改善。
- **run_id 単位のリプレイ可能トレース出力（HTML / JSON）**: 事後に「なぜそう動いたか」を共有・レビュー。
- **実行前の人間向けプラン表示**: `run_test_environment` 等で起動サービス・ネットワーク・破棄対象を読みやすく提示。

> 注意: ダッシュボードは localhost 限定＋必要なら認証。外部に晒さない（§2のセキュリティ方針と矛盾させない）。

---

## 9. テスト環境クイックパス

- `run_test_environment`: Compose相当の環境を一括起動。内部でネットワーク作成・ヘルスチェック待機・後片付けを自動化し、各サービスのアクセス先を返す。
- `wait_for_condition`: TCPポート開放 / ログ内文字列 / healthy を条件に待機（タイムアウト付き）。AIによる `sleep 30` 乱用を排除。

## 11. 推奨実装順

| Phase | 内容 | 狙い |
|------|------|------|
| **0** | セキュリティ土台（§2） | 機能ではなく前提。これ無しに他は成立しない |
| **1** | 出力制御（§6）＋ `run_container_and_exec` | トークン削減ROI最大・実装は軽い |
| **2** | 構造化テスト結果 pytest/jest/go（§4） | AI-firstの本丸 |
| **2+** | Edit/Verify 最小編集ループ（§5） | 失敗を"その場で直して再検証"するループを閉じる。Phase 2と並走 |
| **3** | dry_run＋二段階トークン＋承認キューUI（§7・§8） | 実効性あるHITL |
| **4** | `run_test_environment`＋`wait_for_condition`（§9） | `sleep 30`撲滅 |
| 横断 | トークン削減（§3） | 各Phaseに織り込む。特に§3.2はループ体感が最大 |
| 並行 | 可観測性（§8） | まずジャーナル → ダッシュボード → 通知の順 |

**切る / 別リポジトリ**: セマンティック検索・コードグラフ一式、snapshot系、ネットワーク系の大半（§1）。

## まとめ

ドキュメント1の思想（全部見せない / 構造を返す / HITL）を採用し、その思想に照らしてドキュメント1自身の後半（コード理解レイヤー）を切る。
トークン削減は「1回の出力」より**反復ループの差分化**で稼ぎ、人間側は「常時監視」から**通知＋ダッシュボード＋承認キュー**へ移す。
目標は一貫して——**AIが最小限のコンテキストで最大限の判断を行える、安全なテスト実行基盤**。
